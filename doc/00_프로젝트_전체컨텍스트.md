## 1. 프로젝트가 무엇인가

AgileX **Piper 6축 로봇팔**을 위한 **하드웨어 반사(reflex) 시스템**을 **Zybo Z7-20 (Zynq-7000 XC7Z020) FPGA**로 만드는 학습 프로젝트입니다.

핵심 아이디어는 이렇습니다. 평소 로봇 제어는 PC가 합니다. 그런데 위험한 상황(예를 들어 사람이 너무 가까워지거나 센서 임계를 넘는 경우)에서는 PC와 소프트웨어를 기다릴 시간이 없습니다. 그래서 **FPGA 하드웨어 로직이 직접 위험을 판단하고, 마이크로초 단위로 CAN 버스에 안전 명령(비상정지 등)을 끼워 넣어** 로봇을 멈추거나 회피시킵니다. 사람으로 치면 뜨거운 것에 손이 닿았을 때 뇌를 거치지 않고 척수가 손을 빼는 "반사신경"입니다.

**최종 목표(Phase 3):** 이 반사 판단 코어를 **TinyTapeout 실리콘 칩(ASIC)**으로 만들어, 진짜 칩을 로봇 제어 루프 안에 넣는 것입니다.

**문서화 원칙:** 모든 산출물을 .md로 남겨 다음 세션의 컨텍스트로 재투입합니다.

## 2. 3단계 로드맵 + 현재 위치

| 단계 | 내용 | 상태 |
|---|---|---|
| **Phase 1** | FPGA를 HIL 시뮬레이션 루프에 통합 (Gazebo/ROS ↔ 실제 FPGA ↔ CAN) | ✅ **완성** (2026-06-13) |
| **Phase 2** | 반사 로직 개발 (FPGA가 위험 판단 → CAN override) | 🔵 **진행 중** (반사 칩 A 단계 보드 검증 완료) |
| **Phase 3** | 반사 코어를 TinyTapeout ASIC으로 (칩을 루프에 넣기) | ⏳ A→B→C 단계로 접근 중 |

**지금 위치:** Phase 1은 끝났습니다. Phase 2 반사를 개발하면서, 그 반사 코어를 그대로 Phase 3 TinyTapeout 칩으로 가져가기 위해 **칩-친화적 구조(TT 래퍼)로 RTL을 짜고 있습니다.** A→B→C는 그 칩을 향한 점진적 단계입니다(§4).

## 3. 현재 아키텍처 — 데이터가 실제로 어떻게 흐르나 (★정확히★)

지금 보드(02_reflex_system_dev/02_rs_A_can_in_pl, "A 단계")에서 실제로 도는 구조입니다. 이전 문서들이 여기를 자주 틀리게 적었으니 이 절을 기준으로 삼으십시오.

```
평상시 정상 명령:
  PC ──이더넷(UDP)──> PS(lwIP) ──> PS 내장 CAN(XCanPs) ──> 트랜시버 ──> CAN 버스 ──> 로봇
  (★정상 명령은 PS 내장 CAN으로 바로 나갑니다. PL도 CTU도 칩도 안 거칩니다.)

반사(위험 시):
  DIP/위험 디지털 입력 ──> 칩(reflex_core_tt, 판단) ──> action_id/fire(병렬 핀)
                          ──> reflex_action_map(PL) ──> PL CTU CAN ──> 트랜시버 ──> 버스 ──> 로봇
  (★반사만 PL의 CTU CAN을 씁니다. 칩은 "무엇을" 결정하고, PL이 그걸 CAN 프레임으로 만듭니다.)

포즈 피드백:
  로봇 ──> CAN ──> PS가 XCanPs로 수신 ──> 그냥 버림(카운트만, 반사에 안 씀)
  (★현재 칩은 포즈를 받지 않습니다. 반사는 포즈가 아니라 디지털 위험신호로만 판단합니다.)
```

- **PS 송신과 CTU 송신은 트랜시버 바로 앞에서 wired-AND**로 합쳐집니다. 비상정지(0x150)는 정상 명령(0x155~0x157)과 **CAN ID가 달라서**, CAN 버스의 중재(arbitration)가 낮은 ID를 알아서 우선 채택합니다. 그래서 별도 먹스(mux)나 게이트가 필요 없습니다. 이걸 "AND + 중재" 방식이라 부릅니다.
- 같은 ID로 정상 명령을 덮어써야 하는 반사(예: 특정 관절을 같은 명령 ID로 다른 값으로)는 그때만 "먹스 + 게이트" 층을 추가합니다. 자세한 비교는 [02_reflex_system_dev/README.md](../02_reflex_system_dev/README.md).

## 3-A. 설계 원칙 — 안정된 인터페이스 경계 (★이 프로젝트의 뼈대★)

이 프로젝트가 잘 굴러가는 핵심 이유는 한 가지 설계 원칙을 여러 층에 반복해서 적용했기 때문입니다. **양쪽이 합의한 좁고 안정된 인터페이스(계약)를 경계에 두고, 그 경계 뒤의 내부 구현은 자유롭게 갈아끼운다.** 경계만 안 깨면 한쪽을 바꿔도 다른 쪽은 그대로입니다.

이 프로젝트에 실제로 박혀 있는 경계들입니다(전부 같은 원리).

| 경계 (무엇 ↔ 무엇) | 인터페이스(계약) | 경계 뒤에서 자유롭게 바뀌는 것 | 덕분에 가능한 것 |
|---|---|---|---|
| HILS ↔ FPGA | 이더넷 위 **13바이트 UDP 패킷** (192.168.1.10:5000) | FPGA 내부 설계 전체 (01 / A / B / C) | HILS를 안 고치고 반사 설계를 바꿔가며 **같은 시뮬로 검증**. FPGA를 블랙박스로 봄 |
| 정상 명령 ↔ 반사 | **CAN ID 분리** (0x150 vs 0x155~7) + wired-AND 중재 | 반사 로직·트리거 | 정상 경로(PS 내장 CAN)를 안 건드리고 반사를 **공존**시킴 |
| 반사 로직 ↔ AXI 통신 | **5신호 계약** (reflex_core ↔ axil_master_wr: `wr_req/addr/data` ▶, ◀ `wr_busy/done`) | reflex_core의 판단 로직(무엇을 언제) | AXI를 모르는 채 반사 로직만 키움. AXI 마스터는 **재사용** |
| 반사 블록 ↔ 블록디자인 | reflex_top의 **고정 포트** (m_axi / aclk / aresetn) + 설계별 입력 | 어느 설계든 같은 자리에 결선 | bd 결선을 안 바꾸고 **설계 폴더만 갈아끼움** |
| 칩 ↔ PL | **동결된 TT 핀맵** (SPI 설정 + action_id/fire 병렬 + CAN tx/rx 예약) | 칩에 CAN을 얼마나 넣나 (A→B→C) | CAN-in-chip 결정이 바뀌어도 **핀아웃 재설계 불필요** |

**왜 이게 중요한가 (설계 경험으로 남길 것):**
1. **변경 격리** — 한 층을 바꿔도 경계 건너편은 안 깨진다. 고장이 나면 원인이 경계 안쪽으로 좁혀져 디버깅이 쉽다.
2. **독립 검증** — 각 층을 따로 시뮬·테스트할 수 있다(경계의 계약만 지키면). 예: 반사 로직은 5신호 계약만 맞추면 AXI 없이 sim 가능.
3. **병렬·점진 개발** — 팀원(PC측)과 나(FPGA측)가 13바이트 UDP만 합의하고 따로 개발했다. A→B→C도 핀맵만 동결하면 칩 내부를 단계적으로 키울 수 있다.
4. **재사용** — axil_master_wr, reflex_top 골격, HILS, PL 베이스가 설계마다 그대로 재사용된다.

**주의(경계의 양면성):** 경계는 양쪽이 **명시적으로 적어두고 둘 다 지켜야** 효력이 있다. 한쪽이 경계를 침범하면 깨진다 — 실제로 2026-06-15에 반사 설계를 바꾸면서 설계무관이어야 할 HILS를 건드린 실수가 있었다(메모리 `hils-design-separation`). 그래서 경계마다 "여기는 안 건드림" 표시를 문서·Makefile 상단에 박아 둔다. 또 경계를 너무 좁게/넓게 그으면 오히려 제약이 되므로, 무엇을 고정하고 무엇을 자유롭게 둘지는 신중히 정한다(예: TT 핀맵은 CAN tx/rx를 미리 **예약**해 둬서, CAN-in-chip 여부가 나중에 바뀌어도 핀맵이 안 흔들리게 했다).

각 계약의 상세: 13바이트 UDP는 [03_hils 핸드오프](../01_piper_fpga_testbed/03_hils/piper_pc_handoff/HANDOFF.md)·`CAN-이더넷_브리지_계약서.md`, 5신호 계약·reflex_top은 [02_reflex_system_dev/README.md](../02_reflex_system_dev/README.md)와 [docs/README.md](../02_reflex_system_dev/docs/README.md)·`05_학습자료/11_..wrapper_아키텍처.md`, TT 핀맵 동결은 메모리 `tt-partition-plan`.

## 4. 반사 칩 A→B→C 계획 (TinyTapeout로 가는 길) — ★간소화 반영판★

칩에 무엇을 넣느냐를 한 번에 다 바꾸지 않고, **위험 요소를 한 단계에 하나씩만** 바꿔서 고장 위치를 격리합니다. 이게 단계를 나누는 유일한 이유입니다. 격리되는 새 위험이 없는 작업은 군더더기이므로 뺍니다.

핵심 전제: **칩과 PL을 잇는 SPI는 "규칙 설정·프로그래밍" 전용**입니다. 어느 위험을 어느 동작에 매핑할지, 우선순위, 켜고 끔을 SPI로 씁니다. **포즈 같은 실시간 스트림을 흘리는 통로가 아닙니다.** 그래서 포즈 기반 반사는 칩이 CAN을 직접 받는 C에서만 의미가 있습니다.

| 단계 | 칩이 하는 일 | CAN 위치 | 포즈 | 상태 |
|---|---|---|---|---|
| **A** | 판단만 (action_id 출력). 칩 CAN 핀은 예약만 | PL(CTU) 송수신 | 안 씀(디지털 위험만) | ✅ **보드 검증 완료** |
| **B** (얇게) | 판단 + **작은 Verilog CAN 송신 코어로 직접 송신** | 송신=칩, 수신·디코드=PL | 안 씀 | 다음 |
| **C** (끝모습) | 판단 + CAN 송신 **+ 수신·디코드** | 송수신 다 칩 | **칩이 CAN RX로 직접 수신** | 최종 |

- **A (완료):** TinyTapeout 반사 코어(`reflex_core_tt_v2`, 프로그래밍 가능한 다중 반사 + 우선순위 중재)를 PL에 올려, 실제 Zybo 보드에서 DIP를 올리면 0x150 비상정지가 나가는 것을 검증했습니다(2026-06-15). 01 모놀리식 버전과 바이트·주기까지 동일. **A에서는 포즈 기반 반사를 만들지 않습니다**(칩이 포즈를 안 받으니 억지로 PL→SPI로 포즈를 밀어넣는 가짜 배관이 되고, C에서 칩이 CAN을 직접 읽는 순간 전부 폐기되므로 군더더기). A의 남은 선택 작업은 SPI로 규칙 하나를 바꿔 쓰는 작은 데모(="프로그래밍 가능"이라는 A의 유일한 미증명 주장 확인)뿐이고, 이것도 생략 가능합니다.
- **B (얇게):** B와 C를 통틀어 진짜 모르는 단 하나는 "Verilog로 짠 CAN 코어가 TT 타일 예산(타일당 약 320 플립플롭, 최대 약 20타일)에 들어가는가"입니다. 그래서 **보드에 올리기 전에 먼저 ① TT 합성을 흘려 타일 수를 측정하고(게이트), ② 시뮬로 검증**합니다. 마침 PL에 검증된 CTU CAN이 있으니, 칩의 CAN 송신 코어를 CTU와 버스 상대로 맞붙여 시뮬할 수 있습니다. 시뮬과 크기가 통과하면, B를 **아주 얇게** 한 사이클만 보드에 올립니다 — 칩이 비상정지 한 프레임을 **실제 버스에 직접 송신**(수신·디코드는 PL에 그대로 유지). 칩의 CAN 코어가 실버스에 처음 비트를 쏘는 것을 포즈·수신과 분리해서 검증하는 게 목적입니다. (로봇 안전상 이 얇은 한 단계는 남깁니다.)
- **C (끝모습):** 칩이 CAN 송신과 수신을 다 합니다. **칩이 자기 CAN 수신 핀으로 로봇 포즈를 직접 받아** 판단합니다(SPI 아님). 포즈 기반 반사는 여기서 개발합니다. PL의 CTU는 제거하거나, 테스트베드에서는 비교용으로 켜 둘 수 있습니다.

자세한 핀맵 동결·검증 4단계(HILS→2FPGA→칩)는 메모리 `tt-partition-plan` 참조.

## 8. 핵심 기술 사실 (확정값, 참고용)

**클럭:** PS_CLK 33.333MHz, IO PLL 1000MHz, FCLK0(PL 시스템 클럭) 50MHz. CAN ref 목표 100MHz(`CAN_CLK_CTRL=0x00100A01` = IO PLL ÷10).

**CAN 비트타이밍 (can_clk=100MHz):** 1Mbps → BRPR=9, TS1=6, TS2=1, SJW=1 (BTR=0x10284184). 125kbps → BRPR=79. `bit_rate = 100MHz / ((BRPR+1)×(1+(TS1+1)+(TS2+1)))`, 샘플포인트 80%. XCanPs 레지스터는 실제값-1.

**드라이버/주소:** PS CAN0 base `0xE0008000`(드라이버 XCanPs, SDT 플로우). PL CTU CAN-FD APB base **`0x43C00000`**, TX_COMMAND 오프셋 0x074. 비상정지 트리거 데이터 0x202(txcr bit1 | txb2 bit9). 비상정지 프레임 ID 0x150, B0=01.

**핀 (Pmod JE):** CAN phy_tx → V12(JE1), phy_rx → W16(JE2), LVCMOS33. DIP SW0 → G15. 트랜시버: FPGA phy_tx → 모듈 실크"RX"(실제 TXD), phy_rx → 실크"TX"(실제 RXD) — 모듈 실크가 반대로 인쇄돼 있음.

**★Vivado 2025.2 ps7_init 버그:** BD/XCI가 IO PLL·100MHz로 맞아도 ps7_init이 `0xF8000128 ← 0x00700F01`(÷105, =9.524MHz) 디폴트를 고집함. 앱이 부팅 시 SLCR로 `0x00100A01` 직접 써서 교정하는 전제로 진행. 상세: `02_zybo_skeleton/.../Vivado_ps7_init_버그_분석.md`.

**★DDS 공유메모리 누적 함정:** Docker `--ipc host`라 컨테이너 /dev/shm = 호스트 /dev/shm. Fast-DDS `fastrtps_*` 세그먼트가 root 소유로 쌓여 Gazebo 컨트롤러가 죽음("첫 run만 됨"의 진짜 원인). HILS 실행 스크립트가 매 실행마다 컨테이너 root로 청소해 해결. 메모리 `dds-shm-cruft` 참조.

**TinyTapeout 제약:** 고정 IO 24개(ui_in 8 / uo_out 8 / uio 양방향 8) + clk + rst_n + ena. 타일당 약 320 플립플롭, 최대 약 20타일. Sky130 PDK, Verilog. 이 IO 개수가 진짜 제약(타일을 늘려도 IO는 고정). CTU(VHDL·APB·풀 FD)는 칩에 직접 못 넣음 → 칩용은 작은 Verilog CAN 코어를 새로 짬.

**환경:** Vivado 2025.2 (lin64), roundServer(Ubuntu 24.04)에서 직접 실행. 보드 Zybo Z7-20.
