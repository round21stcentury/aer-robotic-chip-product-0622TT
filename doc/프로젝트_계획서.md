## 2. 시스템 개요

### 2.1 핵심 아이디어
로봇 암의 일반 제어 신호는 호스트 PC에서 생성하지만, **위험 상황에 대한 즉각 반응(반사)** 은 PC나 PS를 거치지 않고 FPGA의 PL에서 센서 입력을 직접 받아 하드웨어적으로 판단·발화한다. 이를 통해 소프트웨어 제어 루프 대비 압도적으로 낮은 지연으로 안전/회피 동작을 수행하는 것이 목표다.

### 2.2 신호 흐름 (데이터 경로)
```
[호스트 PC]
   │  CAN 제어 신호 생성
   │  (Ethernet)
   ▼
[Zybo Z7-20 : PS (ARM Cortex-A9), rtos 기반]
   │  Ethernet으로 제어 신호 수신
   │  (AXI 등 내부 인터커넥트)
   ▼
[Zybo Z7-20 : PL (FPGA Logic)]
   │  ← 센서 입력을 직접 수신 (PS 경유 안 함)
   │  반사 발화 여부 판단 로직
   │  PS의 제어 신호 + 반사 결정을 결합
   ▼
[최종 CAN 신호 스트리밍 출력 (PL에서 직접 제어)]
   │
   ▼
[Piper 로봇 암]
```

### 2.3 역할 분담
| 구성요소 | 역할 |
|---|---|
| 호스트 PC | 상위 제어 신호(궤적/CAN) 생성, 시뮬레이터 구동 |
| Zybo PS | Ethernet으로 제어 신호 수신, PL로 전달, 비실시간 관리 |
| Zybo PL | 센서 직접 입력, 반사 판단, **최종 CAN 스트리밍 제어** |
| (최종) ASIC | PL 로직 중 반사 핵심부를 대체/검증 |

## 3. 기술 스택 / 하드웨어

- **타깃 보드**: Digilent **Zybo Z7-20** (Xilinx Zynq-7000, XC7Z020)
  - PS: Dual-core ARM Cortex-A9
  - PL: Artix-7 등가 FPGA (≈85K logic cells, 220 DSP, BRAM 등)
  - Gigabit Ethernet (PS GEM)
- **로봇**: AgileX **Piper** 6축 로봇 암 (CAN 버스 제어)
- **통신**:
  - PC ↔ Zybo: **Ethernet**
  - Zybo ↔ 로봇: **CAN** (PL에서 스트리밍 제어)
- **HDL / 설계**: [미정 — Verilog/SystemVerilog 가정], Xilinx Vivado
- **ASIC**: **Tiny Tapeout** (오픈소스 MPW shuttle), 최대 **20 타일** 구매 가정
- **시뮬레이션 환경**: **Gazebo** (확정). Piper의 ROS(noetic) 공식 지원과 정합. ROS ↔ Gazebo ↔ HIL 연동.
- **센서**: [미정 — 후보: 힘/토크, 근접, 전류, IMU, 촉각 등. 반사 트리거 물리량 정의 필요]

## 4. 개발 단계 (3-Phase 로드맵)

### 1단계 — FPGA 시뮬레이션(HIL) 환경 통합
> 목표: 시뮬레이션된 로봇과 **실제 FPGA 하드웨어**를 한 루프로 묶는 Hardware-in-the-Loop 환경 구축.

1. **Piper 시뮬레이션 환경 구축**
   - 시뮬레이터 선정 및 Piper 모델 구동
   - PC에서 CAN 제어 신호 생성 → 시뮬레이터 반영 확인
2. **Zybo Z7-20 CAN 스트리밍 환경 구축**
   - PL에서 CAN 프레임을 생성·스트리밍하는 기본 파이프라인 구현
   - PS ↔ PL 데이터 전달(AXI) 경로 확보
3. **FPGA 경유 폐루프 구성**
   - Piper 시뮬레이션 구동 시 신호를 Zybo로 전송 → Zybo에서 처리 후 다시 시뮬레이터로 반환
   - 즉, **시뮬레이터 ↔ Ethernet ↔ PS ↔ PL ↔ (CAN/loopback) ↔ 시뮬레이터** 경로 검증

**1단계 완료 기준**: 시뮬레이션 제어 신호가 실제 FPGA를 경유해 왕복하며, 지연·정합성이 측정 가능한 상태.

### 2단계 — 반사 시스템 개발
1. 1단계 HIL 환경을 이용해 반사 로직 개발 후 **시뮬레이션 상에서 평가**
   - 센서 입력 정의 → 반사 발화 조건 로직 → CAN 신호 개입(override/injection)
2. 시뮬레이션 통과 시 **실제 로봇 실증**

**2단계 완료 기준**: 정의된 위험 시나리오에서 반사가 목표 지연 내에 발화하고, 실로봇에서 안전 동작 확인.

### 3단계 — ASIC 개발
1. FPGA 로직 중 **반사 핵심부 일부**를 Tiny Tapeout으로 **MPW 제작** (≤20 타일 제약)
2. ASIC 수령 시 시스템 루프에 ASIC을 포함시켜 **실증**

**3단계 완료 기준**: ASIC이 FPGA 내 해당 로직을 대체/병행하여 실제 반사 루프에서 동작.

## 6. 제약 조건 및 주의사항
- **Tiny Tapeout 20타일 제약**: ASIC화 대상은 반사 판단 코어 등 **소형 고정기능 로직**으로 한정. 타일 예산 안에서 들어가도록 일찍부터 로직 규모·I/O를 산정해야 함. (TT는 저속 클럭·제한된 디지털 I/O.)
- **반사 경로의 결정성/저지연이 핵심 가치**: 반사 판단은 PC·PS를 거치지 않고 PL이 센서를 직접 받아 처리해야 함.
- **CAN 출력은 PL에서 최종 제어** (PS의 하드 CAN 컨트롤러 단독 사용이 아님).
- **단계 간 인터페이스 고정**: ASIC화를 염두에 두고 반사 코어의 입출력 인터페이스를 2단계에서 미리 ASIC 친화적으로(단순 디지털, 명확한 핀) 설계.

## 8. 용어집 (Glossary)
- **PS (Processing System)**: Zynq의 ARM 프로세서부. 여기서는 Ethernet 수신·관리 담당.
- **PL (Programmable Logic)**: Zynq의 FPGA부. 센서 직접 입력·반사 판단·CAN 스트리밍 담당.
- **HIL (Hardware-in-the-Loop)**: 시뮬레이션 루프 안에 실제 하드웨어(FPGA)를 포함시키는 검증 방식.
- **반사(Reflex)**: 위험 상황에서 PC/소프트웨어를 거치지 않고 하드웨어가 즉시 수행하는 개입 동작.
- **Tiny Tapeout**: 소규모 디지털 회로를 저비용으로 MPW 제작하는 오픈소스 ASIC 셔틀. 타일 단위 면적 예산.
- **MPW (Multi-Project Wafer)**: 여러 설계를 한 웨이퍼에 함께 제작하는 방식.
