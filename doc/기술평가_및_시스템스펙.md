## A. 프로젝트 평가

### A.1 강점 (타당성 높음)
- **단계 분할이 합리적**: HIL → 시뮬레이션 평가 → 실로봇 → ASIC. 안전 검증을 실물 앞에 두는 정석 순서.
- **ASIC 대상 스코프가 작다**: 반사 판단 코어는 본질적으로 소형 고정기능 디지털 로직 → Tiny Tapeout(이하 TT) 20타일 예산에 **여유 있게** 들어감 (§D 참조). 학습용 ASIC 프로젝트로 난이도/규모 균형이 좋음.
- **학습 가치가 수직적으로 넓다**: 로봇 제어(CAN) → FPGA(PS/PL, AXI, XADC) → HIL → ASIC(RTL→GDS) 전 계층을 한 루프에서 경험.

### A.2 핵심 리스크 / 기술 쟁점 (반드시 인지)

**R1. CAN 버스가 물리적 반응의 병목 — "저지연"의 실체를 정확히 이해해야 함**
- Piper CAN = **1 Mbps**. 표준 CAN 2.0A 프레임(11-bit ID + 8 byte) ≈ 108~130 bit → **프레임당 ~110~130 µs**.
- 즉 반사 *판단*이 0 ns여도, 새 명령을 버스에 실제로 밀어넣는 데 **최소 한 프레임(~130 µs)** 이 걸리고, 그 뒤 모터 드라이버 내부 루프 반응(수 ms)이 더해진다.
- **그럼에도 하드웨어 반사의 이득은 실재한다**: PC를 루프에서 제거하면 OS 스케줄링 + USB-CAN 어댑터 + 이더넷 왕복으로 인한 **수 ms~수십 ms의 지연·지터**가 사라진다. 결정 경로가 µs 단위 + 결정성(deterministic) 확보.
- ⟹ **정직한 목표 설정**: "CAN/액추에이터가 만드는 ~수백 µs~ms 바닥(floor)은 못 넘지만, 소프트웨어 경로의 ms급 지터를 제거하고 결정 지연을 µs로 만든다." 이 프레이밍을 KPI에 반영해야 함(§C.4 지연 예산).

**R2. CAN 다중 마스터 충돌 — 아키텍처로 원천 차단 필요**
- PC도 명령을 내고 PL도 개입하면 같은 버스에 두 마스터 → 중재 충돌/일관성 문제.
- **해결책(본 문서가 채택)**: **PL을 로봇에 대한 유일한 CAN 마스터**로 두는 **CAN 게이트웨이 구조**. PC 명령은 Ethernet→PS→PL로 들어와 PL이 **재발행**하고, 반사 발생 시 PL이 명령을 **override/inject**. (계획서의 "최종 CAN 신호는 PL에서 스트리밍 제어"와 정확히 일치.) → §B.2.

**R3. 센서 미정이 반사 설계를 막는 임계 경로**
- 트리거 물리량(접촉력? 근접거리? 토크/전류 급변?)이 정해져야 임계값·필터·발화조건·인터페이스가 결정됨.
- **권장**: 아날로그 근접/힘 센서 → **Zynq 내장 XADC(1 MSPS, 12-bit)** → PL 직결. 보드 내장 자원이라 추가 ADC 없이 µs급 샘플링, "PL이 센서를 직접 받는다"는 요구와 정합 (§C.3, §F).

**R4. Tiny Tapeout 턴어라운드 & 통합 난이도 (3단계 일정 리스크)**
- TT는 셔틀 마감 → 제조 → 배송까지 **수개월~1년** 소요. 3단계는 장기 과제로 잡아야 함.
- TT 칩은 캐리어 보드(RP2040 mux) 뒤에 위치하고 **I/O가 8 in / 8 out / 8 bidir = 24핀**, 클럭 ~50 MHz 상한. → ASIC 코어는 **단순 직렬 인터페이스**로 분할하고 Zybo와 Pmod로 연결하는 통합 설계를 2단계부터 미리 고려.

**R5. CAN 트랜시버 비내장**
- Zybo Z7-20에 CAN 트랜시버 없음. **외장 3.3V CAN 트랜시버**(예: SN65HVD230) 필요, Pmod로 PL 핀에 연결.

### A.3 종합 판정
**기술적으로 타당하고 학습 목적에 매우 적합.** 단, (1) "저지연"의 실효 범위를 CAN/액추에이터 바닥 기준으로 정직하게 설정하고, (2) CAN 게이트웨이 구조로 다중 마스터를 차단하며, (3) 센서를 조기 확정하는 것이 성공의 3대 전제. ASIC 스코프는 TT 20타일에 충분히 수렴하므로 최종 목표(ASIC 포함 루프) 달성 가능성 높음.

## B. 시스템 아키텍처 정의

### B.1 전체 블록도
```
┌─────────────┐  Ethernet(UDP)   ┌──────────────────── Zybo Z7-20 ────────────────────┐
│  호스트 PC   │ ───────────────▶ │  PS (Cortex-A9, RTOS)                                │
│ - 시뮬레이터 │ ◀─────────────── │   - UDP 송수신 / 명령 파싱 / 상태 텔레메트리          │
│ - 궤적/명령  │   상태·텔레메트리 │   - AXI로 PL과 통신 (비실시간)                        │
└─────────────┘                  │            │ AXI4-Lite / AXI-Stream                  │
                                 │            ▼                                         │
   ┌──────────┐  아날로그        │  PL (FPGA)                                           │
   │  센서     │ ───────────────▶│   ┌──────────┐  ┌───────────────┐  ┌─────────────┐  │
   │ (근접/힘) │   → XADC(1MSPS) │   │ Reflex    │  │ Command       │  │ CAN MAC     │  │
   └──────────┘                  │   │ Core ★    │─▶│ Arbiter/Mux   │─▶│ (TX/RX,1Mbps)│──┼─▶ CAN 트랜시버 ─▶ Piper
                                 │   │(임계/FSM) │  │(override/inj) │  │             │◀─┼── (외장 Pmod)  ◀─ 로봇
                                 │   └──────────┘  └───────────────┘  └─────────────┘  │
                                 │     ★ = 3단계 ASIC(Tiny Tapeout) 대체 대상           │
                                 └──────────────────────────────────────────────────────┘
```

### B.2 CAN 게이트웨이 구조 (핵심 결정)
- **PL = 로봇에 대한 유일한 CAN 마스터.** PC/PS는 로봇 버스에 직접 쓰지 않는다.
- 정상 흐름: PC 명령 → PS(UDP) → PL `Command Arbiter` → `CAN MAC`가 로봇에 프레임 발행.
- 반사 흐름: `Reflex Core`가 발화 → `Arbiter`가 PC 명령을 **무시/대체(override)** 하거나 안전 프레임을 **삽입(inject)** → 즉시 CAN 발행. PC 경유 없음.
- 장점: 다중 마스터 충돌 원천 차단, 반사 우선순위 하드웨어 보장, ASIC화 경계가 깔끔(`Reflex Core` 단독 분리).

> **HIL × Docker**: 호스트(24.04)에서 Gazebo/ROS2를 Docker로 돌려도 FPGA HIL에 제약 없음. FPGA는 컨테이너 밖 외부 하드웨어이고, 컨테이너를 `--network host`로 띄우면 UDP(→Zybo)·SocketCAN·DDS가 호스트와 동일하게 동작(NAT 무, 지연 영향 무). SocketCAN 커널 모듈만 호스트에서 `modprobe`. 셋업: [piper-hil-docker/](piper-hil-docker/).

### B.3 PS / PL 기능 분할
| 계층 | 담당 | 비실시간/실시간 |
|---|---|---|
| PS (RTOS) | UDP 통신, 명령 디코드, 파라미터(임계값) 설정, 텔레메트리, 로깅 | 비실시간(soft) |
| PL | XADC 수집, 반사 판단(FSM/임계), 명령 중재, CAN MAC 송수신 | **실시간(hard)** |
| ★ Reflex Core | 센서값 → 발화 판단 (3단계에 ASIC으로 외부화) | 실시간 |

> PS는 사용자가 명시한 **RTOS 기반**(예: FreeRTOS on Cortex-A9). 반사 경로는 PS를 거치지 않으므로 RTOS 지터가 반사 지연에 영향 없음 — 좋은 설계.

### C.4 반사 지연 예산 (Latency Budget) — KPI
| 구간 | 추정 | 성격 |
|---|---|---|
| 센서 아날로그 정착 | 센서 의존 (µs~수십 µs) | 센서 선정 후 확정 |
| XADC 변환 | ~1 µs (1 MSPS) | 고정 |
| **반사 결정 로직** | **< 100 ns** (수 클럭 @100 MHz) | **우리가 최적화하는 핵심** |
| 명령 중재 + CAN 프레임 조립 | < 1 µs | |
| CAN 전송(1프레임) | **~110~130 µs** | **버스 바닥(개선 불가)** |
| 모터 드라이버 내부 반응 | ~1~5 ms | 로봇 내부(개선 불가) |
| **(참고) 소프트웨어 경로 제거 효과** | **수 ms~수십 ms 지터 제거** | **본 프로젝트의 실질 이득** |

> **KPI 제안**: ① 센서→CAN 프레임 송출 시작까지 **결정+전송 지연 < 150 µs**, ② 지터(σ) **< 5 µs**, ③ 소프트웨어 경로 대비 지터 1~2 order 감소를 정량 입증.

### C.5 PL 모듈 정의
1. **`xadc_if`** — XADC 샘플 스트림(1 MSPS), 채널 선택, 디지털 필터(옵션, 이동평균).
2. **`reflex_core`** ★ — 임계 비교 + 발화 FSM(히스테리시스/디바운스 포함). 입력: 센서값·파라미터. 출력: `fire`, `safe_payload`. **3단계 ASIC 대상.**
3. **`cmd_arbiter`** — PC명령 vs 반사명령 mux. 우선순위: 반사 > PC. override/inject 모드.
4. **`can_mac`** — CAN 2.0 비트스트림 송수신(스터핑, CRC, ACK, 중재), 1 Mbps, 로봇 프레임 ID 맵.
5. **`axi_bridge`** — PS↔PL 레지스터/스트림.

## D. Tiny Tapeout ASIC 스펙 (3단계)

### D.1 ASIC 대상 = `reflex_core` 단독
- 이유: 고정기능·저복잡도·실시간 핵심 + 칩 경계가 깔끔(센서값 in → fire/payload out).
- CAN MAC·AXI는 ASIC화하지 않음(I/O·복잡도 과다). PL에 유지.

### D.2 20타일 예산 분석
- TT 타일 ≈ **~1,000 표준셀/타일**(셔틀별 변동). **20타일 ≈ ~20,000 셀** 예산.
- `reflex_core` 추정 규모: 비교기 + 파라미터 레지스터 + 디바운스/히스테리시스 FSM + 직렬 I/O ≈ **수백~수천 셀**.
- ⟹ **20타일 내 여유 충분.** 오히려 여유 타일에 다채널 센서 융합/필터(이동평균·중앙값)·자가진단을 넣어 학습 가치를 키울 수 있음.

### D.3 I/O 매핑 (TT 고정: 8 in / 8 out / 8 bidir + clk + rst)
| 핀 그룹 | 할당(예시) |
|---|---|
| `ui_in[7:0]` | 센서 디지털값 직렬 입력 + 모드 비트 |
| `uo_out[7:0]` | `fire`, 상태/진단, `safe_payload` 직렬 출력 |
| `uio[7:0]` | Zybo PL과 SPI류 핸드셰이크(파라미터 로드, 동기) |
| `clk` | Zybo 공급(≤50 MHz, 권장 10~25 MHz) |
| `rst_n` | Zybo 공급 |
> 24핀 제약 → 센서값·페이로드는 **직렬화** 전제. 파라미터(임계값)는 부팅 시 SPI로 로드.

### D.4 TT ↔ Zybo 통합
- TT 캐리어 보드 I/O ↔ **Zybo Pmod** 연결(3.3V 정합 확인).
- Zybo PL은 TT ASIC을 **외부 peripheral**로 구동: 클럭·리셋·파라미터 공급, `fire`/payload 수신 → `cmd_arbiter`로 전달.
- 검증: 동일 입력에 대해 **PL `reflex_core`(RTL) 결과 == TT ASIC 결과** 비교(골든 모델 대조).

### D.5 ASIC 흐름 메모
- RTL(SystemVerilog) → TT 템플릿 → OpenLane(sky130) → GDS → 셔틀 제출. Wokwi 시뮬은 ≤100 kHz 권장.
- 2단계의 `reflex_core`를 **TT 친화 코딩 규약**(단일 클럭, 동기 리셋, 직렬 I/O, 외부 메모리 없음)으로 작성해두면 3단계 이식 비용 최소화.

## E. 단계별 구체화 (Deliverable + 검증)

### 1단계 — HIL 환경
- **D1-1** 시뮬레이터(**Gazebo + ROS noetic**) + Piper 모델 구동, PC가 CAN 명령 생성 → 시뮬 반영.
- **D1-2** PL `can_mac` + `axi_bridge` 구현, PS UDP↔AXI 경로, 외장 CAN 트랜시버로 실제 프레임 송출.
- **D1-3** HIL 폐루프: `시뮬(PC) → UDP → PS → PL → CAN 트랜시버 →(루프백 트랜시버)→ PL → PS → UDP → 시뮬`.
  - 초기엔 PL 내부 루프백으로 단순화 → 이후 물리 CAN 루프백.
- **검증**: 왕복 지연/지터 측정, 프레임 무손실, 시뮬 상태 일관성.

### 2단계 — 반사 시스템
- **D2-1** 센서 확정 → `xadc_if` + `reflex_core`(TT 친화 규약) + `cmd_arbiter` 구현.
- **D2-2** HIL에서 위험 시나리오(접촉/근접 침범) 주입 → 반사 발화·override 평가(§C.4 KPI).
- **D2-3** KPI 통과 시 실로봇 실증(저속·소프트 토크 안전모드부터).
- **검증**: 발화 정확도(오발화/미발화율), 지연/지터, 실로봇 안전 정지/회피.

### 3단계 — ASIC
- **D3-1** `reflex_core`를 TT(sky130, ≤20타일) 제출, 골든 모델 대조 테스트벤치 완비.
- **D3-2** 칩 수령 후 Pmod 통합, PL-RTL 결과와 비트 단위 대조.
- **D3-3** ASIC을 반사 경로에 포함시켜 HIL→실로봇 재실증.
- **검증**: ASIC==RTL 동치, 통합 루프 지연/안전 동작.
