## 0. 한 줄 요약

당신(ROS/PC 담당)이 만들 것 = **시뮬의 Piper 명령을 이더넷으로 FPGA에 보내고, FPGA가 실제 CAN으로 되쏜 걸 다시 받아 루프를 닫는 PC 측 절반.** FPGA 측 절반(UDP→CAN 컨트롤러)은 상대(FPGA 담당)가 구현합니다. 경계면은 **이더넷 위 13바이트 UDP 패킷 1개**뿐.

## 2. 확정된 결정 (이대로 구현, 바꾸지 말 것)

1. **반환 경로 = (b) 실제 CAN + USB-CAN.** FPGA가 실제 CAN으로 되쏘고, USB-CAN(`can0`)에서 SocketCAN으로 직접 받음. → **UDP 5001 수신부 구현 안 함.**
2. **전송 = UDP** (TCP 금지). 프레임 1개 = 패킷 1개, 즉시 전송(버퍼링 금지).
3. **패킷 = 정확히 13바이트**: `can_id`(빅엔디언 4B) + `dlc`(1B) + `data[8]`. payload는 **재해석 없이 그대로** 통과. (브리지 계약서 §2)
4. **IP/포트 확정값**: FPGA `192.168.1.10`, PC `192.168.1.100`, 명령 포트 **UDP 5000**. (브리지 계약서 §1)
5. **CAN 버스**: 클래식 CAN 2.0(FD 아님) / 11비트 표준 / **1 Mbps** / 종단저항 120Ω 양끝 / GND 공통.
6. **1차 프레임 범위만 구현**: 명령 0x151+0x155~0x157+0x471, 피드백 0x2A1+0x2A5~0x2A7. (MIT·Cartesian·미확인 ID 제외 — 프로토콜 계약서 상단 ⚠️ 박스)

## 3. 권장 토폴로지 (1안 — piper_sdk 안 건드림)

명령이 **이더넷+FPGA를 반드시 거치게** 하려면 컨트롤러가 실제 `can0`에 바로 쓰면 안 됩니다. 각 엔드포인트는 인터페이스 1개만:

```
[컨트롤러]  piper_sdk, vcan0 하나만 (TX/RX 둘 다)        ✅ SDK 단일 인터페이스 유지
   │ 명령 TX → vcan0
   ▼
[브리지]   vcan0의 명령 프레임 → UDP 5000 → FPGA → 물리 CAN TX
                                                 │  USB-CAN can0 수신(+HW ACK)
                                                 ▼
[가상로봇] can0에서 명령 디코드 → Gazebo 구동 → 피드백 생성 → vcan0에 씀
   │
   ▼
[컨트롤러] vcan0에서 피드백 RX                            ✅
```
- 컨트롤러=vcan0 하나, 가상로봇=can0 하나 → **둘 다 단일 인터페이스라 SDK 패치 불필요.**
- 명령은 FPGA를 거쳐야만 can0에 나타남. 피드백은 FPGA 안 거치고 vcan0 직행(FPGA는 명령 경로 전용).
- vcan0↔can0 ID 필터 라우팅: `cangw`(can-utils) 또는 직접 작성한 ROS 노드.
- 자세한 그림·근거: 브리지 계약서 **§7.1**.

## 4. ★당신(팀원)이 결정·구현할 열린 항목★

1. **3절의 "브리지 + 가상로봇 노드"를 어떻게 구현할지** — `cangw` 조합이냐, 단일 ROS 노드냐 등.
2. **가상로봇이 piper_sdk의 역방향**(명령 디코드 + 피드백 인코드)을 어디까지 구현할지. piper_sdk는 "로봇에게 말하는" 클라이언트라 "로봇 노릇"은 직접 짜야 함.
3. 인터페이스 이름(`vcan0`/`can0`)은 권장값 — 환경 맞춰 조정 가능하되 **양쪽이 합의**.

> 난이도는 'SDK 패치'가 아니라 '노드 작성' 수준 (1안 덕분).

## 5. 단계적 브링업 (한 번에 다 닫지 말 것)

1. **transport+HW만**: 0x155 한 프레임 → vcan0 → 브리지 → FPGA → `can0`에서 `candump`로 **바이트 그대로(엔디언/13바이트) 도착** 확인. (가상로봇 불필요)
2. **레이턴시 측정** (왕복 타임스탬프). ※ gs_usb USB 왕복 지연 탓에 편도 <1ms를 못 맞출 수 있음 → **결함 아님, 측정 대상.**
3. **가상로봇** 붙여 Gazebo 루프 닫기.

---

## 6. 통합을 조용히 깨뜨리는 3대 함정 (꼭 확인)

1. **CAN FD vs 클래식** — 둘 다 클래식 2.0. 한쪽이 FD면 안 붙음.
2. **엔디언** — `can_id` 빅엔디언(`00 00 01 23` = 0x123). 안 맞추면 ID 뒤집힘.
3. **TCP 사용** — 지연 지터로 HIL 붕괴. 무조건 UDP.

## 7. 환경 사실 (실측 확인됨)

- 시뮬(piper_ros) 기본 CAN 인터페이스 = **`can0`** (`can_port` ROS 파라미터로 변경 가능)
- USB-CAN = **gs_usb** 모듈, **1 Mbps**, SocketCAN
- Docker `network_mode: host` (호스트 UDP/SocketCAN 그대로)
- 근거: `01_piper_sdk/piper-hil-docker/` (`can_activate.sh`, `docker-compose.yml`, `piper_ctrl_single_node_new.py`)
- 기준 SDK: **piper_sdk v0.6.1 / 프로토콜 V2** (펌웨어 V1.5-2+ 필요 — 실기 펌웨어 버전 확인해 공유)
