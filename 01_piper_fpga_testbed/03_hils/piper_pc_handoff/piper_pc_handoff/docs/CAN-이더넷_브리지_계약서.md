# CAN ↔ 이더넷 브리지 인터페이스 계약서

> HIL(Hardware-in-the-Loop) 구성에서 **ROS 시뮬레이터(PC)** 와 **Zybo FPGA** 를 잇는
> CAN-over-UDP 브리지의 규격. 양쪽이 이 문서대로 각자 절반을 구현하면 합쳤을 때 동작한다.
>
> - **팀원(ROS 담당)** 이 구현할 절반: `ROS ↔ CAN프레임 ↔ UDP(이더넷)`
> - **나(FPGA 담당)** 가 구현할 절반: `UDP(이더넷) ↔ Zynq CAN 컨트롤러`
> - **계약 경계면 = 이더넷 위의 UDP 패킷 1개.** 이 바이트 레이아웃만 맞으면 양쪽은 서로 몰라도 됨.

---

## 0. 토폴로지 & 책임 분담

```
        ┌──────────────────── PC (팀원 담당) ────────────────────┐         ┌──── Zybo (나 담당) ────┐
        │                                                        │         │                         │
[ROS 시뮬레이터] ──CAN프레임──▶ [브리지: CAN→UDP] ──UDP/이더넷──▶ │ ──▶ [PS: UDP→CAN컨트롤러] ──▶ CAN TX
        ▲                                                        │         │                         │
        │                                                        │         │                         ▼
        └────────────── (반환 경로 — §7에서 택1) ◀───────────────┴─────────┴──── [CAN 트랜시버] ────┘
```

- **명령 경로 (sim → FPGA):** 이더넷/UDP. ← 팀원의 핵심 산출물(브리지)
- **반환 경로 (FPGA → sim):** ✅ **(b) 실제 CAN + USB-to-CAN 으로 확정** (2026-06-13). FPGA CAN TX → 트랜시버 → USB-CAN(can0) → sim SocketCAN. 상세는 §7.

> ✅ 윗줄(명령) = `ROS→vcan→브리지→UDP→FPGA`, 아랫줄(반환) = `FPGA→실제CAN→USB-CAN→sim`.
> **권장 토폴로지(SDK 안 건드리는 1안)는 §7.1 참조** — 각 엔드포인트는 인터페이스 1개만 쓰고, 라우팅은 HIL 노드가 담당.

## 1. 전송 계층 (반드시 지킬 것)

| 항목 | 규격 | 이유 |
|---|---|---|
| 프로토콜 | **UDP** (TCP 금지) | TCP는 재전송·Nagle로 지연 지터 발생 → HIL 타이밍 붕괴 |
| 송신 방식 | **프레임 1개 = 패킷 1개, 즉시 전송** | 묶어 보내면 지연/지터. 버퍼링·딜레이 금지 |
| FPGA IP | `192.168.1.10` (고정) | DHCP 금지, 정적 IP |
| PC IP | `192.168.1.100` (고정) | 〃 |
| 포트 sim→FPGA | UDP **5000** | 명령 경로 |
| 포트 FPGA→sim | ~~UDP 5001~~ **미사용** | 반환을 (b) 실제 CAN으로 확정 → 이더넷 반환 안 씀 |
| MTU | 패킷 ≤ 64바이트 | 단일 CAN 프레임이라 작음. 단편화 없음 |

> ✅ **위 값은 확정값(2026-06-13).** 그대로 사용. 네트워크 환경 때문에 바꿔야 하면 양쪽 동시에 이 표를 갱신.
> 반환 경로를 (b)로 확정했으므로 **UDP 5001 수신부는 구현하지 않는다.** 반환은 USB-CAN(can0)에서 SocketCAN으로 직접 받는다(§7).

## 2. 패킷 바이트 레이아웃 ★가장 중요★

**한 패킷 = CAN 프레임 1개 = 13바이트 고정.** (여기서 통합이 거의 다 깨짐 — 1바이트 단위로 박아둠)

| 오프셋 | 크기 | 필드 | 설명 |
|---|---|---|---|
| 0 | 4 | `can_id` | **빅엔디언(네트워크 바이트오더).** 11비트 표준 ID를 하위 11비트에 배치. 상위 비트 0 |
| 4 | 1 | `dlc` | 데이터 길이 0~8 |
| 5 | 8 | `data[8]` | 페이로드. **바이트 그대로 통과(재해석 금지).** 안 쓰는 바이트는 0 |

```
바이트:  [ ID3 ID2 ID1 ID0 ] [ DLC ] [ D0 D1 D2 D3 D4 D5 D6 D7 ]
          └─ 빅엔디언 32bit ─┘  1B          8B (payload, 그대로)
```

### 엔디언 (틀리면 ID가 뒤집혀 엉뚱한 프레임)
- `can_id` 필드는 **빅엔디언**. 예) ID `0x123` → 바이트 `00 00 01 23`.
- 페이로드(`data`)는 **브리지가 절대 건드리지 않음.** 바이트 순서 해석은 시뮬레이터와 로봇의 몫(Piper는 빅엔디언). 브리지는 **투명한 바이트 운반기**일 뿐.

## 3. CAN 규격 (로봇 Piper에 고정) — 틀리면 조용히 안 붙음

| 항목 | 값 |
|---|---|
| 프레임 종류 | **클래식 CAN 2.0 (CAN FD 아님!)** |
| ID 형식 | **11비트 표준** (확장 29비트 아님) |
| 비트레이트 | **1 Mbps** |
| 종단저항 | **120Ω 양 끝** (USB-CAN쪽 + 트랜시버쪽) |
| 접지 | **GND 공통** |

> ⚠️ 한쪽이 CAN FD로 열거나 비트레이트 다르면 물리적으로 안 붙음. 둘 다 **클래식 2.0 / 1Mbps**.

---

## 4. 타이밍 예산

- **프레임당 패킷 1개, 생성 즉시 송신** (큐잉·배칭 금지).
- 한쪽 방향 목표 지연: **< 1 ms**, 지터 최소화.
- 시뮬레이터 제어 주기에 송신 주기를 맞춤. (Piper 제어 주기 확정값은 `CAN_프로토콜_계약서.md` 참조)

---

## 5. 유실 / 순서 정책

- UDP라 **유실 가능** → HIL 테스트에선 허용. **재전송 안 함**(최신 명령이 곧 도착하므로).
- 순서 역전 대비: 필요하면 패킷에 1바이트 시퀀스 번호를 추가하는 옵션(기본은 안 씀). 추가 시 §2 레이아웃을 14바이트로 갱신하고 양쪽 합의.

## 6. 양쪽 의사코드

### 팀원 쪽 (PC, 파이썬 + SocketCAN)
```python
import socket, struct
sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
FPGA = ("192.168.1.10", 5000)

# CAN → UDP (sim에서 나온 프레임을 FPGA로)
def can_to_udp(can_id, data: bytes):
    dlc = len(data)
    pkt = struct.pack(">I", can_id)            # 빅엔디언 4B
    pkt += bytes([dlc])                         # DLC 1B
    pkt += data.ljust(8, b'\x00')              # data 8B (패딩)
    sock.sendto(pkt, FPGA)                      # 즉시 전송

# (반환경로 (b) 확정 → 아래 udp_to_can 은 미사용. 반환은 can0 SocketCAN에서 직접 수신)
# def udp_to_can(pkt: bytes):  # (a) 이더넷 대칭일 때만 필요했음 — 지금은 불필요
#     can_id = struct.unpack(">I", pkt[0:4])[0]
#     dlc    = pkt[4]
#     data   = pkt[5:5+dlc]
#     return can_id, data
```

### 나(FPGA) 쪽 (Zynq PS, lwIP UDP + XCanPs)
```c
// UDP 수신 콜백: 패킷 → CAN 컨트롤러로 송신
void on_udp_rx(u8 *pkt, u16 len) {
    u32 can_id = (pkt[0]<<24)|(pkt[1]<<16)|(pkt[2]<<8)|pkt[3];  // 빅엔디언
    u8  dlc    = pkt[4];
    u32 Tx[4];
    Tx[0] = XCanPs_CreateIdValue(can_id, 0, 0, 0, 0);
    Tx[1] = XCanPs_CreateDlcValue(dlc);
    Tx[2] = (pkt[5]<<24)|(pkt[6]<<16)|(pkt[7]<<8)|pkt[8];       // data[0..3]
    Tx[3] = (pkt[9]<<24)|(pkt[10]<<16)|(pkt[11]<<8)|pkt[12];    // data[4..7]
    XCanPs_Send(&Can, Tx);
}
// 반대로 CAN 수신 시 위 13바이트로 직렬화해 UDP 5001로 송신
```

## 7. 반환 경로 — ✅ (b) 실제 CAN + USB-to-CAN 으로 확정 (2026-06-13)

FPGA의 **진짜 CAN 송신 하드웨어를 실제로 검증**하는 게 본 목적이라 (b)로 확정.
(CAN TX 하드웨어는 2026-06-11에 Zybo↔USB-CAN 1Mbps 양방향까지 이미 검증됨.)

- 반환 경로 = **FPGA CAN TX → 트랜시버 → USB-CAN(can0) → sim SocketCAN**
- 따라서 **UDP 5001 / 이더넷 대칭 반환은 구현하지 않음.**

### 7.0 SocketCAN 인터페이스 사실 (시뮬 측, 실측 확인)
- 시뮬(piper_ros) 기본 CAN 인터페이스 = **`can0`** (`can_port` ROS 파라미터로 변경 가능)
- USB-CAN = **gs_usb** 모듈, **1 Mbps**, SocketCAN
- Docker는 `network_mode: host` → 호스트의 UDP/SocketCAN을 NAT 없이 그대로 사용
- 근거: `01_piper_sdk/piper-hil-docker/` (`can_activate.sh`, `docker-compose.yml`, `piper_ctrl_single_node_new.py`)

### 7.1 ★권장 토폴로지 (1안) — piper_sdk 안 건드림★

명령은 반드시 이더넷+FPGA를 거쳐야 하므로(프로젝트 목적), 컨트롤러가 실제 `can0`에 바로 쓰면 안 된다.
대신 **각 엔드포인트는 인터페이스 1개만** 쓰고(=SDK 단일 인터페이스 모델 유지), 라우팅은 HIL 노드가 담당한다:

```
[컨트롤러]  ← piper_sdk, vcan0 하나만 사용(TX/RX 둘 다 vcan0)   ✅ SDK 안 깨짐
   │ 명령(0x15x,0x471) TX → vcan0
   ▼
[브리지]  vcan0에서 명령 프레임만 읽음 → UDP 5000 → FPGA → 물리 CAN TX
                                                        │
                                          USB-CAN can0  │ 수신 (+ 하드웨어 ACK)
                                                        ▼
[가상로봇 노드]  can0에서 명령 읽고 디코드 → Gazebo 구동 → 피드백(0x2A5 등) 생성
   │ 피드백을 vcan0에 씀
   ▼
[컨트롤러]  vcan0에서 피드백 RX   ✅
```
- **컨트롤러**는 `vcan0` 하나만, **가상로봇**은 `can0` 하나만 본다 → 둘 다 단일 인터페이스라 SDK 패치 불필요.
- 명령은 **FPGA를 거쳐야만** can0에 나타남 → 이더넷+FPGA가 진짜 경로에 들어감.
- 피드백은 FPGA를 안 거치고 vcan0로 직접 → **FPGA는 "명령 경로"에만** (반사가 명령을 override하는 원칙과 일치, §10).
- 물리 버스 ACK = FPGA TX ↔ USB-CAN (2026-06-11 검증된 구성).
- vcan0/can0 사이 ID 필터 라우팅은 `cangw`(can-utils) 또는 직접 작성한 ROS 노드로.

### 7.2 ★팀원이 결정·구현할 열린 항목★
- **위 1안의 "브리지 + 가상로봇 노드"를 어떻게 구현할지** (cangw 조합 vs 단일 ROS 노드 등).
- 가상로봇이 piper_sdk의 **역방향(명령 디코드 + 피드백 인코드)**을 어디까지 흉내낼지. piper_sdk는 "로봇에게 말하는" 클라이언트라 "로봇 노릇"은 직접 구현 필요.
- (난이도는 'SDK 패치'가 아니라 '노드 작성' 수준 — 1안 덕분에 내려감.)

### 7.3 단계적 브링업 (전체를 한 번에 닫지 말 것)
1. **transport+HW 무결성만**: 컨트롤러가 0x155 한 프레임 → vcan0 → 브리지 → FPGA → `can0`에서 `candump`로 **바이트 그대로(엔디언/13바이트) 도착** 확인. (가상로봇 불필요)
2. **레이턴시 측정** (왕복 타임스탬프). ※ gs_usb USB 왕복 지연 때문에 §4의 편도 <1ms를 못 맞출 수 있음 → 이건 결함이 아니라 **측정 대상**.
3. 그 다음 **가상로봇** 붙여 Gazebo 루프 닫기.

## 8. 통합 전 체크리스트 (합치기 전에 각자 확인)

- [ ] 둘 다 **클래식 CAN 2.0 / 1Mbps / 11비트** 로 설정했는가
- [ ] 패킷이 **정확히 13바이트**, `can_id`가 **빅엔디언**인가
- [ ] IP/포트 **확정값**으로 §1 표 갱신했는가
- [ ] **UDP**인가 (TCP 아님)
- [ ] 브리지가 payload를 **재해석 없이 통과**시키는가
- [ ] 종단저항 120Ω 양끝, GND 공통인가
- [x] 반환 경로 = **(b) 실제 CAN + USB-CAN 으로 확정** (UDP 5001 미구현)
- [ ] 명령 TX는 **vcan0**, 반환 RX는 **can0** 로 인터페이스 분리했는가 (§7.1)

## 9. 통합을 조용히 깨뜨리는 3대 함정 (팀원에게 강조)
1. **CAN FD vs 클래식** — 한쪽이 FD면 안 붙음. 둘 다 클래식 2.0.
2. **엔디언** — `can_id` 바이트오더 안 맞추면 ID 뒤집힘.
3. **TCP** — 쓰면 지연 지터로 HIL 붕괴. 무조건 UDP.

## 10. 아키텍처 주의 (미리 못 박기)
- 이 브리지 경로는 **비-실시간 명령용**이다.
- 나중에 들어갈 **반사(reflex) 로직은 FPGA 내부에서 끝난다.** 이 이더넷 왕복에 **절대 의존하지 않는다.**
- 팀원이 "모든 제어를 이 브리지로 한다"고 설계하면 반사 단계에서 꼬임 → **"명령=브리지, 반사=FPGA 내부"** 분리 원칙 공유할 것.
