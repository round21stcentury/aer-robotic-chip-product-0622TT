# 스텝1 「MCP 제어기 실증 — 정상명령 패스스루」 설계 및 디버깅 보고서

> 폴더: `02_1_reflex_system_TT/01_mcp_제어기_실증/`
> 작성일: 2026-06-18
> 한 줄 요약: **PC 로봇의 정상 관절명령이 FPGA(PS→PL→SPI→TinyTapeout 칩→MCP2515→CAN)를 그대로 통과(패스스루)해 로봇을 움직이는지**를 실증하는 단계. 스텝2~4(반사)를 얹기 위한 **먹스(mux) 골격**을 이 단계에서 미리 심어 둔다.

---

## 1. 이 단계의 목적과 위치

### 1.1 목적
TinyTapeout 칩(이하 "칩")을 최종적으로 **CAN 버스의 유일한 송신자**로 만들 계획이다. 그렇다면 평상시 로봇을 움직이는 **정상 관절명령조차도 반드시 칩을 통과**해야 한다(칩이 송신을 독점하므로). 스텝1은 바로 이 **정상명령 통과 경로(패스스루)** 가 하드웨어 실물에서 동작함을 증명하는 단계다.

또한 스텝2~4에서 반사가 끼어들 때 **정상명령을 끊고 반사명령을 주입**할 자리(먹스)가 필요하다. 이 먹스를 **스텝1부터** 칩 안에 넣어 둔다. 스텝2~4는 먹스의 선택신호(`reflex_active`)만 올리면 된다 — 나중에 경로를 다시 뜯어고치지 않기 위함이다.

### 1.2 전체 HIL(Hardware-In-the-Loop) 루프에서의 위치
```
[PC]                                  [Zybo FPGA 보드]                              [PC]
Gazebo 슬라이더 ── 이더넷 UDP ──▶ PS(ARM, lwIP) ── AXI GPIO ──▶ PL(로직)
                  (13바이트/프레임)                              │ SPI 3.125MHz
                                                                ▼
                                                       TinyTapeout 칩(PL 안에 인스턴스)
                                                                │ SPI 6.25MHz
                                                                ▼
                                                          MCP2515 (Pmod JE)
                                                                │ CAN-H/L 1Mbps
                                                                ▼
USB-CAN(can0) ◀──────────────────────── 물리 CAN 버스 ─────────┘
   │
   ▼
virtual_robot(--cmd-iface can0) ──▶ Gazebo 로봇 (슬라이더 따라 움직임)
```
- **상류(PC→보드):** Gazebo의 관절 슬라이더가 만든 CAN 명령을 이더넷으로 보드에 던진다.
- **보드 내부:** PS(소프트웨어)는 받아서 전달만 하고, **실제 중계 일은 PL+칩(하드웨어)** 이 한다.
- **하류(보드→PC):** 칩이 MCP2515를 통해 물리 CAN 버스에 명령을 실어 보내면, USB-CAN으로 PC가 받아 `virtual_robot`이 Gazebo 로봇에 적용한다.

이 토폴로지는 03_hils의 계약(13바이트 UDP 패킷, CAN id 규약, 1Mbps)을 그대로 따른다. 즉 03_hils 시뮬레이터를 한 줄도 안 고치고 그대로 쓴다.

## 2. 계층별 설계 — 무엇을 만들었고, 서로 무슨 데이터를 주고받는가

네 개의 계층(PS 소프트웨어 / PL 로직 / 칩 / MCP2515)과 그 사이의 세 개 경계(이더넷, AXI GPIO, SPI 두 종)로 나눠 설명한다.

### 2.1 계층 ① — PS 소프트웨어 (ARM Cortex-A9, lwIP 이더넷 브리지)
**파일:** `fpga/src/reflex_s1_main.c`

**역할:** 이더넷으로 들어온 CAN 명령을 받아 PL에게 넘기는 **얇은 다리(bridge)**. 반사 판단 같은 지능은 일절 없다(설계 원칙: 반사·중계는 하드웨어, PS는 설정·전달만).

**구현:**
1. **네트워크 기동:** lwIP(lwip220) 위에서 UDP 포트 5000을 연다. 보드 IP는 `192.168.1.10`, MAC `00:0a:35:00:01:02`, PHY 100Mbps 고정. (05_rs_xadc_trigger에서 검증된 lwIP 설정 재사용.)
2. **수신 콜백 `udp_recv_cb`:** 13바이트 패킷이 오면 파싱한다.
   - 바이트 0~3 = CAN id(빅엔디안), 바이트 4 = DLC, 바이트 5~12 = 데이터 8바이트.
   - id가 `0x150`~`0x157` 범위면 **id별 최신값 테이블**(`tbl_lo[8]`, `tbl_hi[8]`)에 저장하고 `tbl_dirty` 플래그를 세운다. 즉, **같은 관절명령이 여러 번 와도 가장 최신 한 개만 기억**한다(밀린 옛 명령을 버려 지연을 막음).
3. **메인 루프 — 레이트 페이싱:** dirty한 프레임을 **라운드로빈으로 한 번에 하나씩** PL에 넘기고 `PACE_US`(기본 150µs) 쉰다. 칩이 한 프레임을 MCP로 보내는 데 ~128µs가 걸리므로, 그보다 느리게 보내야 칩이 프레임을 떨어뜨리지 않는다.
4. **설정 적재:** 부팅 때 `cfg` GPIO에 `(enable<<8)|SPI_DIV`를 써서 칩→MCP SPI 속도와 enable을 정한다.
5. **관측 출력:** 약 1초마다 EMIO GPIO로 칩이 되읽은 MCP 상태(CANSTAT/CNF/EFLG/TEC 등)와 수신·송신 카운터를 시리얼로 찍는다.

### 2.2 경계 A — PS ↔ PL : AXI GPIO 메일박스 + EMIO 관측
**정의:** `fpga/vivado/add_reflex_s1.tcl`

PS가 PL에게 데이터를 넘기는 통로다. lopper(Vitis 디바이스트리 생성기)가 한글경로·커스텀 AXI 슬레이브·입력 GPIO에 약하므로, **출력은 표준 axi_gpio, 입력은 PS7 EMIO GPIO**만 쓴다(검증된 안전 조합).

| 신호 | 방향 | AXI 주소 | 폭 | 내용 |
|---|---|---|---|---|
| `cfg_gpio` | PS→PL | `0x41200000` | 16 | `[7:0]`=칩→MCP SPI 분주, `[8]`=enable |
| `cmd_lo` | PS→PL | `0x41210000` | 32 | 정상프레임 데이터 D0~D3 (D0=`[7:0]`) |
| `cmd_hi` | PS→PL | `0x41220000` | 32 | 정상프레임 데이터 D4~D7 |
| `cmd_id` | PS→PL | `0x41230000` | 32 | `[31]`=토글 비트, `[10:0]`=CAN id |
| `obs0/obs1` | PL→PS | EMIO bank2/3 | 64 | MCP 되읽기(관측) |

**메일박스 프로토콜:** PS가 한 프레임을 넘길 때 `cmd_lo`, `cmd_hi`를 먼저 쓰고 **마지막에 `cmd_id`의 토글 비트를 뒤집는다**. PL은 토글이 바뀌는 순간을 "새 프레임 도착"으로 인식한다.

### 2.3 계층 ② — PL 로직 (chip_feeder + PL측 SPI 마스터)
**파일:** `pl/rtl/reflex_top_s1.v`(최상위), `pl/rtl/chip_feeder_s1.v`(공급기), `pl/rtl/spi_master.v`(칩 대상 SPI 마스터)

**`chip_feeder_s1` 역할:** 메일박스로 받은 정상프레임을 **SPI로 칩의 정상프레임 레지스터(0x50~0x55)에 써넣는** 일과, 주기적으로 **칩을 통해 MCP 상태를 되읽어 관측**하는 일을 한다.

**구현(상태기계):**
1. **부팅:** 칩 레지스터 0x03(SPI_DIV)·0x02(CONTROL)를 적재 → `configured=1`.
2. **운전 루프(`S_CHK`):** 우선순위 = ①새 프레임 릴레이 ②SPI 속도 변경 반영 ③주기적 MCP 되읽기. **로봇 명령(릴레이)을 관측보다 먼저** 처리해 명령이 관측 때문에 늦지 않게 한다.
3. **프레임 릴레이(`S_RLY`):** 한 프레임당 SPI 쓰기 6번 — 0x50(id)→0x51(D1,D0)→0x52(D3,D2)→0x53(D5,D4)→0x54(D7,D6)→0x55(송신 트리거).
4. **★settle 래치(디버깅으로 추가, 9장 참조):** 토글 변화를 감지하면 곧바로 래치하지 않고 `SETTLE_N`(64클럭=1.28µs) 기다린 뒤, **릴레이 중이 아닐 때** `cmd_lo/cmd_hi/cmd_id`를 한꺼번에 래치한다. AXI GPIO 세 슬레이브가 서로 다른 시각에 갱신되는 쓰기 스큐(skew)로 인한 섞임을 원천 차단한다.

**`spi_master`(PL측) 역할:** `chip_feeder`의 요청을 24비트 SPI 트랜잭션 `{rw, addr[6:0], data[15:0]}`(MSB first, Mode 0)으로 칩에 전송. 칩이 내부에서 클럭을 오버샘플하므로 **충분히 느리게**(분주 HALF=8) 보낸다.

### 2.4 경계 B — PL ↔ 칩 : SPI #1 (PL이 마스터, 칩이 슬레이브), **3.125MHz**
- 클럭원 FCLK = 50MHz, 반주기 = HALF(8)×20ns = 160ns → SCLK = **50MHz/(2×8) = 3.125MHz**.
- 칩 핀: `uio[0]`=SCLK, `uio[1]`=MOSI, `uio[2]`=CSN(이상 PL→칩), `uio[3]`=MISO(칩→PL).
- 한 트랜잭션 24비트. 칩 슬레이브가 2단 동기화 + 엣지검출로 오버샘플하므로 이 속도가 안전선(반주기 ≥ 6클럭)을 넘는다.

### 2.5 계층 ③ — TinyTapeout 칩
**파일:** `chip/rtl/tt_um_reflex_s1.v`(최상위), `chip/rtl/spi_slave_full.v`(SPI 슬레이브+레지스터), `chip/rtl/mcp_tx_mux.v`(먹스), `chip/rtl/mcp_init.v`(MCP 초기화), `chip/rtl/mcp_tx_send.v`(프레임 송신), `chip/rtl/mcp_probe.v`(되읽기), `chip/rtl/mcp_arb4.v`(중재기), `chip/rtl/spi_master_mcp_v2.v`(MCP 대상 SPI 마스터)

칩은 표준 TinyTapeout 래퍼(`tt_um_*`: `ui_in[8]`/`uo_out[8]`/`uio[8]`, 24 IO 고정)를 따른다. 내부 블록:

1. **`spi_slave_full` — SPI 슬레이브 + 레지스터 파일:** PL의 SPI 마스터를 받아 레지스터를 읽고 쓴다. **정상프레임 레지스터(0x50~0x55)** 가 이 단계의 핵심이다 — PL이 여기 써넣고 0x55에 쓰면 `norm_send` 1클럭 펄스가 나간다. (반사용 레지스터 0x10~0x45도 있으나 스텝1은 미사용.)
2. **`mcp_tx_mux` — 먹스(게이트):** `sel_send = reflex_active ? reflex_send : normal_send` 식으로 정상명령과 반사명령 중 하나를 고른다. **스텝1은 `reflex_active=0` 하드코딩 → 정상명령이 항상 통과.** 스텝2~4는 이 신호만 살린다.
3. **`mcp_init` — MCP 초기화:** 부팅 시 MCP2515를 리셋하고 CNF1/2/3(8MHz 오실레이터에서 1Mbps: 0x00/0xC0/0x80), 송신버퍼·인터럽트를 설정. **리셋 직후 오실레이터 안정화 지연(`RESET_DELAY`=10ms)** 을 둔다(디버깅으로 추가, 9장).
4. **`mcp_tx_send` — 프레임 송신:** `send` 펄스가 오면 id·데이터를 **원자적으로 래치**한 뒤 MCP 송신버퍼(TXB0)에 12회 SPI 쓰기 + RTS로 한 CAN 프레임을 발사.
5. **`mcp_probe` — 되읽기:** 주기적으로 MCP 상태 레지스터(CANSTAT/CNF/EFLG/TEC/REC 등)를 읽어 슬레이브 0x21~0x28에 채운다(관측용).
6. **`mcp_arb4` — 중재기:** 위 init/tx/probe 세 클라이언트가 **하나의 MCP SPI 드라이버**를 다투지 않게 시간분할 중재.
7. **`spi_master_mcp_v2` — MCP 대상 SPI 마스터:** 실제 MCP SPI 파형 생성. **분주를 런타임 입력(`half_div`)으로** 받아 PS가 재합성 없이 속도를 바꿀 수 있다.

### 2.6 경계 C — 칩 ↔ MCP2515 : SPI #2 (칩이 마스터, MCP이 슬레이브), **6.25MHz**(가변)
- 반주기 = `half_div`(기본 4)×20ns = 80ns → SCLK = **50MHz/(2×4) = 6.25MHz**. PS가 0x03 레지스터로 바꿀 수 있다(예: `make build SPI_DIV=6` → 4.17MHz).
- 칩 핀: `uio[4]`=SCLK, `uio[5]`=MOSI, `uio[6]`=CSN(이상 칩→MCP), `ui_in[3]`=MISO, `ui_in[2]`=MCP 인터럽트.

### 2.7 계층 ④ — MCP2515 → CAN → 로봇
MCP2515(Pmod JE, V12/W16/J15/H15/V13)가 칩이 써넣은 프레임을 물리 CAN-H/L에 1Mbps로 송신. USB-CAN이 `can0`로 받아 `virtual_robot`이 Gazebo 로봇에 적용.

## 3. 데이터 흐름 — 관절명령 한 개의 일생

슬라이더에서 1번 관절을 움직였을 때(CAN id 0x155):
```
① Gazebo 슬라이더 → 03_hils → 13바이트 UDP 패킷
     [00 00 01 55][08][D0 D1 D2 D3 D4 D5 D6 D7]  → 192.168.1.10:5000
② PS udp_recv_cb: id=0x155 → idx=5 → tbl_lo[5]={D3,D2,D1,D0}, tbl_hi[5]={D7,D6,D5,D4}, dirty
③ PS 메인루프(≤150µs 후): send_mailbox(0x155, tbl_lo[5], tbl_hi[5])
     cmd_lo←데이터, cmd_hi←데이터, (DSB) cmd_id←토글^1 | 0x155
④ PL chip_feeder: 토글변화 감지 → 64클럭 settle → 래치(f_id/f_lo/f_hi)
⑤ PL→칩 SPI 6회: 0x50←id, 0x51~0x54←데이터, 0x55←트리거  →  칩에 norm_send 펄스
⑥ 칩 mcp_tx_mux: reflex_active=0 → 정상통과 → mcp_tx_send가 id·데이터 원자래치
⑦ 칩→MCP SPI 12+1회: TXB0에 적재 + RTS  →  MCP이 CAN 프레임 발사
⑧ CAN 1Mbps → USB-CAN can0 → virtual_robot → Gazebo 1번 관절이 슬라이더 따라 이동
```
**핵심:** ②~⑤ 어디서도 데이터가 깨지지 않도록 ④의 settle과 ③의 쓰기순서(DSB), ⑥의 원자래치가 보장한다(9장에서 검증).

## 4. 칩 SPI 슬레이브 레지스터 맵 (PL↔칩 계약)

| 주소 | 이름 | R/W | 내용 |
|---|---|---|---|
| 0x00 | ID_MAGIC | R | 0xCAFD (칩 식별) |
| 0x01 | VERSION | R | 0x0511 (스텝1 빌드) |
| 0x02 | CONTROL | R/W | `[0]`=enable |
| 0x03 | SPI_DIV | R/W | `[7:0]`=칩→MCP SCLK 반주기 |
| 0x10~0x13 | RULE0~3 | R/W | 반사 규칙(스텝2~4) |
| 0x18~0x1B | THRESH0~3 | R/W | 반사 임계(스텝2~4) |
| 0x30 | XADC_VAL | R/W | 아날로그 트리거값(향후) |
| 0x40~0x45 | RECOIL_DELTA | R/W | 움츠림 델타(스텝4) |
| **0x50** | **NORM_ID** | R/W | **정상프레임 CAN id `[10:0]`** |
| **0x51~0x54** | **정상 데이터** | R/W | **{D1,D0}/{D3,D2}/{D5,D4}/{D7,D6}** |
| **0x55** | **NORM_SEND** | W | **쓰면 정상프레임 송신 1펄스** |
| 0x20 | STATUS | R | `[2]`=init_done, `[1]`=reflex_active |
| 0x21~0x28 | MCP 되읽기 | R | CANSTAT/CANCTRL/CNF1/2/3/EFLG/{TEC,REC}/CANINTF |

**0x50~0x55(굵게)가 이 단계에서 신설한 정상명령 패스스루의 핵심**이다.

## 5. 통신 속도 정리 (질문하신 항목)

| 링크 | 마스터 | 슬레이브 | 속도 | 결정 방식 |
|---|---|---|---|---|
| PL ↔ 칩 (SPI #1) | PL | 칩 | **3.125 MHz** | 합성 고정(HALF=8). 칩 오버샘플 안전선 ≥ 6클럭 충족 |
| 칩 ↔ MCP (SPI #2) | 칩 | MCP2515 | **6.25 MHz** (기본) | **PS 런타임 가변**(0x03 레지스터, `make build SPI_DIV=n`) |
| MCP ↔ 로봇 (CAN) | — | — | 1 Mbps | CNF1/2/3 = 0x00/0xC0/0x80 (8MHz 오실레이터) |

**TinyTapeout 고려:** 칩은 "PL의 슬레이브이면서 동시에 MCP의 마스터"라는 이중 역할을 가진다. 위 두 SPI 속도가 그 두 역할에 각각 대응한다. 칩을 PL 안에 인스턴스로 품은 채(`reflex_top_s1`이 `tt_um_reflex_s1`을 직접 인스턴스화) 설계해, 나중에 칩을 실물 TinyTapeout으로 떼어내도 핀맵(24 IO)이 그대로 맞도록 했다.

## 6. TinyTapeout 24핀 핀맵 (칩 경계 동결)

| 핀 | 방향 | 용도 |
|---|---|---|
| `uio[0]` | 입력 | PL→칩 SPI SCLK |
| `uio[1]` | 입력 | PL→칩 SPI MOSI |
| `uio[2]` | 입력 | PL→칩 SPI CSN |
| `uio[3]` | 출력 | 칩→PL SPI MISO |
| `uio[4]` | 출력 | 칩→MCP SPI SCLK |
| `uio[5]` | 출력 | 칩→MCP SPI MOSI |
| `uio[6]` | 출력 | 칩→MCP SPI CSN |
| `ui_in[2]` | 입력 | MCP 인터럽트 |
| `ui_in[3]` | 입력 | 칩→MCP SPI MISO |
| `ui_in[7]` | 입력 | arm_enable |
| `uo_out[0]`,`uo_out[7]` | 출력 | init_done 등 상태표시 |

반사 트리거(DIP 스위치/FSR 센서)는 스텝2~4에서 남은 `ui_in`에 배정한다.

## 7. 빌드·실행 자동화 (Makefile)
`make sim`(RTL 시뮬) → `make fpga`(합성+비트+XSA) → `make build`(PS 앱) → `make program`(보드 다운로드) → `make can-up`+`make can-mon`(HIL 관측). 단축어: `serial`/`can-mon`/`program`/`can-up`/`rerun`. **주의:** Vivado는 `-jobs 2` 고정(8은 14GB RAM에서 OOM), Vitis 워크스페이스는 ASCII 경로(`_vitis_ws/s1`)에 둔다(한글경로서 lopper 깨짐).

## 8. 검증 현황
- **RTL 시뮬:** 칩 통합(`tt_um_reflex_s1`)·PL 통합(`reflex_top_s1`) 패스스루 시뮬 PASS.
- **★스트림 디버그 시뮬(`pl/sim/tb_stream.v`, 9장):** 155/156/157을 연속·스큐·연달아 보내도 **데이터 섞임 0** → RTL 데이터 경로 무손상 증명.
- **HIL 실측(통제된 입력):** 이더넷 연결·MCP 초기화(CANSTAT=0x00, CNF=00/C0/80, TEC=0) 확인. **★패스스루 바이트 단위 완벽 검증:** 50Hz·100Hz 다중 id 손상 0·드롭 0, 1번 관절 매끄러운 스윕 깨끗 추종(9.5절).
- **★전체 루프 실로봇 검증:** Gazebo 로봇이 슬라이더를 따라 움직임 확인. ★자율 검증: 1번 관절 0→20° 스윕 주입 → 로봇 피드백(0x2A5)이 19.9° 추종(루프 닫힘). 컨트롤러 명령 스트리밍 완전성: FPGA 전달집합 = 브리지 명령집합(COMMAND_IDS_WITH_ESTOP)과 정확히 일치 → 누락 0 (10장).

## 9. 디버깅 여정 — 무엇을 의심했고, 무엇을 고쳤고, 무엇이 유령이었나

이 단계에서 가장 많은 시간을 쓴 부분이라 상세히 기록한다.

### 9.1 증상의 변천
1. **초기:** 로봇이 슬라이더를 따라가긴 하나 **버벅이고 반응이 늦음**.
2. **악화:** **팔을 마구 휘저어 분해된 것처럼** 보임(violent flailing).
3. **최종:** **아예 반응을 안 함**.

### 9.2 의심과 수정 (시간순)
| # | 의심한 원인 | 적용한 수정 | 결과 |
|---|---|---|---|
| 1 | `chip_feeder`가 릴레이 도중 새 프레임을 래치해 한 프레임 안에서 데이터가 섞임 | 릴레이 중(`S_RLY`/`S_RLY_W`)에는 래치 금지 | 호전, 그러나 잔존 |
| 2 | PS가 `cmd_lo/hi`(데이터)보다 `cmd_id`(토글)를 먼저 PL에 도달시켜, PL이 새 id+옛 데이터를 래치 | `send_mailbox`에서 데이터 쓰기 뒤 **DSB 메모리배리어** → 토글이 항상 데이터보다 나중 도착 | 큰 섞임 감소 |
| 3 | AXI GPIO 세 슬레이브의 쓰기 스큐가 여전히 남음 | PL에 **settle**: 토글 감지 후 64클럭 기다려 데이터 안정 후 래치 | 재합성·프로그램했으나 **여전히 섞여 보임** |

### 9.3 유령의 정체 — 핵심 깨달음
수정 3까지 했는데도 candump에 `00 00 6D B9` 옆에 `00 00 6D 4C`가 보여 "바이트 단위 섞임이 남았다"고 판단하고 칩·SPI까지 의심을 확장했다. **이것이 잘못된 판단이었다.**

- `0x6DB9`=28089, `0x6D4C`=27980. 차이는 **109(=0.109°)**. 이것은 섞임이 아니라 **슬라이더가 0.1° 움직인 정상 연속값**이다.
- 또한 `FF FF 48 62`가 0x155·156·157 모두에 보인 것도 — 1·3·5번 관절이 같은 각도(-47.5°)에 있을 때 나타나는 정상값일 수 있다.
- 사용자께서 "0은 슬라이더 초기값이다, 왜 신경쓰는지 모르겠다"고 정확히 지적하신 대로, **데이터 값 자체는 정상이었다.**

### 9.4 결정적 검증 — 스트림 디버그 시뮬 (`pl/sim/tb_stream.v`)
유령 추적을 멈추고, **연속 스트림 + 최악 쓰기 스큐**를 시뮬로 재현해 RTL이 정말 섞는지 직접 확인했다. 각 프레임의 데이터에 자기 id를 박아 넣고, MCP로 나간 프레임의 id와 데이터가 일치하는지 자동 검사했다.
- **스큐 6클럭 + 정상 간격:** 22프레임 검사, **섞임 0**.
- **연달아(간격 30클럭, 페이싱 실패 모사):** **섞임 0** (settle이 최신 프레임만 일관되게 래치, 중간 프레임은 드롭).

**결론: RTL 데이터 경로(chip_feeder→SPI→칩→MCP)는 어떤 타이밍에서도 데이터를 섞지 않는다.** settle 수정이 옳았다. 디스크의 비트스트림도 settle 소스보다 나중에 빌드된 것을 타임스탬프로 확인.

### 9.5 ★최종 규명 — 실측으로 잡은 진짜 원인 (TXB0 덮어쓰기 splice)
보드 직접 제어 권한을 받아, **입력을 완벽히 통제한 실측**으로 원인을 확정했다(추측 종료).

**(1) 손상은 FPGA가 주입한다 — 입력/출력 동시 비교:**
- `vcan0`(FPGA 입력, PC가 보내는 것) = `00 00 00 00 FF FF 79 A5` **완벽한 상수, 한 글자도 안 틀림.** PC 무죄.
- `can0`(FPGA 출력, 칩이 내보낸 것) = 70% 정확, 30% 손상. → **손상은 PS·PL·칩에서 발생.**

**(2) 손상의 정체 — splice(이어붙임):** 세 id(0x155/156/157)에 ★서로 구별되는 패턴★을 실어 보내니 명백해졌다. 손상 프레임은 **"앞 k바이트=자기 프레임, 뒤 (8−k)바이트=다음 프레임"** 형태. 예: 0x155 가 `11 22 33 44 | EE FF 99 00`(앞 4=155, 뒤 4=156의 꼬리). **다른 관절끼리 섞인 게 아니라(그건 settle 이 막음), 시간상 연속된 두 프레임이 이어붙은 것.**

**(3) 원인 — MCP 송신 중 TXB0 덮어쓰기:** 칩이 한 프레임을 MCP 송신버퍼(TXB0)에 적재하고 RTS(송신요청)한 뒤, **MCP 이 그 프레임을 CAN 버스에 바이트씩 직렬화하는 ~130µs 동안**, 다음 프레임이 와서 칩이 **아직 안 나간 TXB0 뒷바이트를 덮어쓰면**, MCP 이 그 뒷바이트를 다음 프레임 값으로 내보낸다 = splice.

**(4) 레이트 의존의 증명:**
- 단일 id 는 **100Hz 도 손상 0** (연속 프레임 데이터가 같아 덮어써도 안 보임).
- 다중 id(155/156/157)를 **몰아 보내면 손상**, **6.6ms 간격으로 띄우면 손상 0.**
- ∴ 손상 = 프레임이 MCP 송신(130µs)보다 촘촘히 적재될 때만.

**(5) 정밀 원인 — 페이싱 < 파이프라인:** 메일박스→릴레이(46µs)→칩 TXB0 적재(46µs)→MCP 송신(130µs) = **파이프라인 222µs**. PS 페이싱이 150µs 였으므로 다음 프레임 적재가 이전 송신 끝(222µs)보다 일찍(196µs) 시작 → **끝 26µs 겹침 → 끝 몇 바이트 splice.** 관측된 "끝바이트 손상"과 정확히 일치.

**(6) 해결 — 페이싱을 파이프라인보다 크게:** `PACE_US` **150 → 300µs**. 다음 TXB0 적재가 이전 MCP 송신이 끝난 뒤 시작 → 겹침 0. (칩 재합성 불필요, PS 앱 ELF 만 재빌드.)

**(7) 검증 (전용 도구 `fpga/tools/hil_corruption_test.py`):**
- 50Hz·100Hz 다중 id 버스트: **손상 0, 드롭 0** (출력 수 = 입력 수 정확히).
- 1번 관절 0→30000→0 매끄러운 스윕: **깨끗하게 추종, 비정상 튐/splice 0, 2번관절 오염 0.** ← 사용자가 직접 했던 "1번만 천천히" 시나리오 재현, 완전 해결.

> ※ 칩에 "RTS 후 TXB0CTRL.TXREQ 가 0 될 때까지 폴" 코드를 넣었으나, 이 MCP/배선에선 TXREQ 가 0 으로 읽혀 폴이 즉시 통과(무효). 그래서 페이싱으로 해결. **더 견고한 후속안:** mcp_tx_send 에 RTS 후 ~140µs 고정 지연을 넣어 칩이 PS 타이밍과 무관하게 스스로 덮어쓰기를 막게 함(스텝2~4 재합성 때 통합).

### 9.6 정정 — 틀렸던 중간 가설들
정직하게 기록한다. 디버깅 중 다음을 잘못 짚었다:
1. **"바이트 섞임"을 손상으로 오인** — 초기 candump 의 `6D B9`→`6D 4C` 는 손상이 아니라 슬라이더 0.1° 이동(정상값)이었다. (사용자가 "0 신경쓰지 마"로 정확히 지적.)
2. **"무반응은 Gazebo 먹통" 추정** — 실제로는 진짜 FPGA 손상(위 TXB0 덮어쓰기)이 남아 있었다. 통제된 실측 전까지 PC측을 의심한 것은 오판.

### 9.7 교훈
- **관측값을 "손상"으로 단정하기 전, 정상 입력으로 설명되는지 먼저 볼 것**(0.1° 이동 오인).
- **입력을 완벽히 통제한 실측이 추측을 끝낸다** — vcan0/can0 동시 비교로 FPGA 주입을 1분 만에 확정. 구별 패턴(3 id 서로 다른 값)이 splice 메커니즘을 즉시 드러냄.
- **추론이 막히면 시뮬·실측으로 재현**해 가설을 결정적으로 검증.

## 10. HIL 실로봇 구동 — 무동작 원인(Enable)과 명령 스트리밍 완전성

손상(9장)을 고쳐 can0 에 명령이 깨끗이 나오는데도 **Gazebo 로봇이 안 움직이는** 별개 문제가 있었다(2026-06-18 규명).

### 10.1 원인 — Enable 명령(0x471)이 FPGA 전달집합 밖
- `backend_gazebo.set_targets`: `if not self.enabled: return` — **enable 안 된 상태면 관절 명령을 받아도 아무것도 안 함.**
- `self.enabled` 는 `MOTOR_ENABLE = 0x471` 프레임을 받아야 True.
- 그런데 PS 전달조건이 `0x150~0x157` 이라 **0x471 은 범위 밖 → FPGA 가 버림.** 관절(0x155~7)은 통과하지만 로봇은 영원히 disabled → 무동작.
- sim(mock_fpga 는 전 id 전달)은 되고 HIL 만 안 되던 것과 정확히 일치.

### 10.2 해결 — FPGA 전달집합 = 브리지 명령집합과 정확히 일치
브리지(`can_udp_bridge.py`)는 vcan0 에서 `SCOPE_COMMAND_IDS = {0x151,155,156,157,471}`(--estop 시 +0x150)만 하드웨어 CAN 필터로 받아 FPGA 로 보낸다. 그래서 PS 전달집합을 그 최대집합 **`COMMAND_IDS_WITH_ESTOP = {0x150,0x151,0x155,0x156,0x157,0x471}`** 과 정확히 일치시켰다(`FWD_IDS`). → **브리지가 보내는 어떤 명령도 누락되지 않음.** 새 명령 id 추가 시 `FWD_IDS` 에도 반드시 추가(누락=무동작).

### 10.3 검증
- 주입: enable(0x471=FF 02) + 5 명령 id → can0 에 전부 출현, 바이트 일치, 손상 0.
- ★전체 루프: 1번 관절 0→20° 스윕 주입 → 로봇 피드백 0x2A5 가 19.9° 추종(루프 닫힘 자율 증명).
- 스트리밍 완전성: FPGA 전달집합 = 브리지 명령집합 → 누락 0(코드로 확정).

### 10.4 DIP 스위치 — 스텝1엔 기능 없음(정상)
스텝1은 순수 패스스루라 칩에 `reflex_active=1'b0` 하드코딩, XDC 에 MCP 핀만 있고 DIP 핀이 없다. DIP 로 반사를 거는 것은 **스텝2~4 기능.** 스텝1에서 DIP 무반응은 정상.

---

## 11. 다음 단계
스텝1의 패스스루+먹스 골격이 확정되면, 스텝2~4는 **칩 안에서 `reflex_active`만 올려** 정상명령을 끊고 반사(e-stop / 덕 포즈 / 현재포즈 움츠림)를 주입한다. PL·PS·메일박스·SPI 경계는 그대로 재사용한다 — 이것이 먹스를 스텝1부터 심어 둔 이유다.

**견고화 후속:** ① 칩 mcp_tx_send 에 RTS 후 ~140µs 고정지연(PS 페이싱 의존 줄여 칩이 스스로 TXB0 덮어쓰기 차단) — 스텝2~4 재합성 때 통합. ② `FWD_IDS` 는 브리지 명령집합과 동기 유지.
